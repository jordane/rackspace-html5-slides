Rackspace Cloud Monitoring

A brief history

  • Cloudkick (CK), a project to provide monitoring and data visualisation
  • CK founded in Jan 2009
  • Acquired by Rackspace in 2010
  • CK evolved into two products, Cloud Monitoring & Control Panel
  • Cloud Monitoring entered Unlimited Availability in 2012

Cloud Monitoring

How does it work?

  • In 5 (soon to be 6) Datacenters:
  • Agent (client-end, runs checks on the client) connects to Agent-End-Point
  • Collectors run server-end checks (ping, http, etc)
  • Collecting DCs are customer choice, but get 3 free in price of each check
  • Data passed to 3 primary DCs: ORD, DFW, LON
  • Data is processed against alarms (code written by customer)
  • DCs quorum for each check (make sure each DC agrees with each other)
  • Check passed to Messengers, who de-dupe and alert the customer

Many Rackspace Products Use CM

But we don't

Why not?

  • A monitoring system can't monitor itself:
  • Break alerts, you'll never get them
  • Break collectors or AEPs, you might never know
  • Break processing of alarms, ...
  • etc

Alternatives

  • Munin
  • Cacti
  • Nagios
  • Graphite
  • StatsD

We chose Nagios & Graphite

  • Munin has huge issues with scalability
  • Cacti is clunky and annoying
  • Nagios is scalable (with passive checks)
  • Graphite is the only real choice for data visualisation
  • Federated Graphite is scalable
  • StatsD doesn't offer much more than Graphite

Nagios

  • Mostly passive checks
  • Checks are run as time-splayed cronjobs on individual servers
  • send_nsca sends check results to nagios servers
  • 3 Nagios servers, one in each primary DC
  • All Nagios servers report all checks, no dedupe
  • Nagios servers only have to parse status file, little overhead

Federated Graphite

  • Stores data in carbon (a derivative of RRD)
  • Accepts most anything without configuration (uses defaults)
  • Federated splits the storage for carbon on multiple servers
  • This gives mostly limitless scalability (removes disk i/o bottlenecks)
